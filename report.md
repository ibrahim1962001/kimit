# Project Data Persistence Report

## 1. المشروع العام
- اسم المشروع: DataPath Analyzer / Kemet AI Studio
- التركيب: واجهة React + TypeScript في `datapath-app`، وواجهة خلفية FastAPI في `backend`
- دعم ثنائي اللغة: العربية و الإنجليزية
- تكامل AI: Groq API مستخدم في الواجهة الأمامية والخلفية
- تكوين استضافة Firebase موجود فقط عبر `firebase.json`، ولكنه لا يحتوي على كود Firebase فعلي في المشروع

## 2. بيانات المستخدم والمحلية المحفوظة
### 2.1. المفاتيح والتخزين في المتصفح
- `localStorage` يستخدم في الواجهة الأمامية لحفظ:
  - `groq_key`
    - موقع الاستخدام: `datapath-app/src/App.tsx`
    - محتوى الحفظ: مفتاح Groq API الذي يدخل المستخدم
  - `datapath_chat_sessions`
    - موقع الاستخدام: `datapath-app/src/components/ChatPanel.tsx`
    - محتوى الحفظ: جلسات الدردشة الحالية، تضم `id`، `title`، `filename`، و `messages`
    - `messages` تحتوي على رسائل المستخدم والمساعد مع الطوابع الزمنية
  - `datapath_chat_history`
    - موقع الاستخدام: `datapath-app/src/components/ChatPanel.tsx`
    - هذه مفتاح ترحيل قديم للاحتفاظ بتاريخ الدردشة السابق

### 2.2. بيانات المستخدم الممكن استخدامها كسياق مستقبلي
- مفتاح API الخاص بـ Groq (`groq_key`) يمكن اعتباره بيانات مستخدم شخصية أو إعدادات جلسة
- محفوظات الدردشة (`datapath_chat_sessions`) هي أهم مصدر لعمل "سياق المستخدم" المستقبلي
- لا يوجد في الكود الحالي حفظ بيانات المستخدم مثل:
  - حساب مستخدم
  - تسجيل دخول/تسجيل مستخدم
  - هوية مستخدم أو ملف شخصي
  - إعدادات مستمرة غير `groq_key`

## 3. بيانات المشروع الأخرى المحفوظة أو المؤقتة
### 3.1. الواجهة الأمامية
- لا يتم حفظ حالة `dataset` في التخزين المحلي مباشرة.
- البيانات المتعلقة بمساحة العمل الحالية (`dataset`, `filters`, `customCharts`) تبقى في الذاكرة فقط أثناء تشغيل التطبيق.
- `DashboardPage` يعيد قراءة `groq_key` من `localStorage` عند إنشاء الملخص التنفيذي AI.

### 3.2. الواجهة الخلفية
- يحتوي `backend/main.py` على:
  - متغير `DATA_STORE = {}`
    - هذا تخزين في الذاكرة للملفات المرفوعة خلال مدة عمل الخادم
    - لا يتم حفظه على القرص أو في قاعدة بيانات ثابتة
  - تحميل متغيرات البيئة عبر `.env`
    - `backend/.env` يحتوي على مفتاح Groq API
  - لا يوجد تخزين مستخدم دائم في `backend` حاليًا

## 4. حالة Firebase الحالية
- الملف `firebase.json` موجود ويشير إلى استضافة `datapath-app/dist`.
- لا يوجد في البحث عن الكود أي استدعاء لـ Firebase SDK أو Firestore أو `firebaseConfig`.
- إذا كان الهدف ربط المشروع بـ Firebase مستقبلًا، فالحالة الحالية هي:
  - إعداد استضافة فقط
  - لا توجد طبقة مصادقة أو قاعدة بيانات Firebase مستخدمة بعد

## 5. توصيات لربط بيانات المستخدم بـ Firebase مستقبلاً
1. استخدام `Firestore` أو `Realtime Database` لتخزين:
   - جلسات الدردشة (`datapath_chat_sessions`)
   - مفتاح API أو إعدادات المستخدم (يشترط تأمينها أو إبقاءها سرية)
   - معلومات المستخدم/الملف الشخصي
2. إضافة طبقة مصادقة Firebase `Authentication` وحقول تعرّف المستخدم.
3. تخزين `groq_key` بشكل آمن في `Firestore` أو `Firebase Functions` مع التشفير المناسب بدلاً من `localStorage` فقط.
4. استخدام `firebase.json` الحالي كقاعدة استضافة، لكن يلزم إضافة إعدادات عميل Firebase في الواجهة الأمامية.

## 6. الخلاصة
- البيانات المحفوظة حالياً في المشروع هي:
  - `groq_key` في `localStorage`
  - `datapath_chat_sessions` و `datapath_chat_history` في `localStorage`
  - بيانات مرفوعة مؤقتة في `backend` عبر `DATA_STORE` في الذاكرة
- لا يوجد تخزين مستخدم حقيقي أو Firebase فعال في الكود الحالي
- الملف `report.md` هذا يوثّق حالة حفظ البيانات الحالية ويجعل من الممكن لاحقًا ربط مخطط "سياق المستخدم" بـ Firebase دون تعديل الكود الأساسي الآن
